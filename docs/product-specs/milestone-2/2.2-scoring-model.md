# PRD 2.2 — Scoring model

This is the part the buddies will rewrite. The naive baseline
below is deliberately simple so they have a clean target to beat
and a clean interface to plug into.

## Public contract

```python
def score_player(
    player: Player,
    stats_history: list[WeeklyStats],   # past weeks of this season + prior season
    league_scoring: ScoringSettings,    # league.scoring_settings dict from Sleeper
    risk: float,                        # 0.0 = max safety, 1.0 = max gamble
) -> PlayerScore:
    """Return mean projected points, variance, and final risk-adjusted score."""
```

`PlayerScore` carries:

```python
class PlayerScore:
    player_id: str
    projected_mean: float
    projected_variance: float        # stddev, in points
    risk_adjusted_score: float       # what we sort by
    confidence: Literal["low", "medium", "high"]
    notes: list[str]                 # human-readable explanations
```

## Why league_scoring is required

Different leagues weight stats differently (PPR vs half-PPR vs
standard, custom bonuses for long TDs, etc.). The model converts
**stat counts** (yards, receptions, TDs) into **points** by applying
`league_scoring` weights per stat code. Sleeper exposes these
weights as a flat dict (`{"pass_yd": 0.04, "rec": 1.0, ...}`)
matching the stat codes in our stats files — so the same projection
mechanism works for every league format without special-casing.

## Naive baseline algorithm (v1)

For each candidate player:

1. **Per-week historical points.** For each week in
   `stats_history`, compute
   `points_w = sum(league_scoring[code] * value for code, value in week.stats.items() if code in league_scoring)`.
   Stat codes missing from `league_scoring` contribute zero.
2. **Sample selection.**
   - If ≥3 weeks of *this season* are available: use this season's
     weeks only.
   - Else: pad with the previous season's per-week points until we
     have at least 4 samples or run out of data.
   - If the player has zero games in either window (rookie, deep
     waiver): `confidence = "low"`, `mean = 0`, `variance = 5.0`
     (placeholder), `notes += "no historical data"`.
3. **Mean = arithmetic mean of `points_w`.**
4. **Variance = sample stddev of `points_w`.** With 1 sample,
   stddev is undefined → fall back to last-season stddev for the
   same position bucket (computed once at startup from the prior
   season's per-game points across all players at that position).
5. **Confidence.**
   - `high` if ≥4 this-season weeks.
   - `medium` if 1–3 this-season weeks (padded with prior season).
   - `low` if zero this-season weeks.
6. **Risk-adjusted score.**
   `score = mean + (risk - 0.5) * 2 * variance`.
   - `risk = 0.5` → score == mean (variance-neutral).
   - `risk = 0.0` → score = mean − variance (penalise uncertainty).
   - `risk = 1.0` → score = mean + variance (reward upside).
7. **Team preference multiplier** is applied by the caller (the
   pipeline), not by `score_player`. Scoring sees only the player
   and stats; preferences are presentation-layer adjustments.

## Worked examples

A WR averaging 8.0 PPG with stddev 3.0 over 4 games:

| risk | score | reasoning |
| -- | -- | -- |
| 0.5 | 8.0 | mean only |
| 0.0 | 5.0 | penalised by 1 stddev |
| 1.0 | 11.0 | rewarded by 1 stddev |

A WR averaging 12.0 PPG with stddev 10.0 over 4 games (boom/bust):

| risk | score | reasoning |
| -- | -- | -- |
| 0.5 | 12.0 | mean only |
| 0.0 | 2.0 | heavy penalty for inconsistency |
| 1.0 | 22.0 | massive upside |

A risk-averse user picks the first WR; a gambler picks the second.

## Where to put a smarter model

`services/decision-engine/src/decision_engine/core/scoring/`:

- `naive.py` — the v1 baseline above. Reference implementation.
- `<your_model>.py` — sibling module, same `score_player` signature.

A config flag (default `"naive"`) selects between them. The CLI
prints which model was used so output is unambiguous, and the
buddies can run two models on the same snapshot and diff the
output.

## Out of scope for the naive model

- Matchup-aware adjustments (opposing defense rank, weather).
- Injury weighting beyond what `injury_status` exposes to the CLI
  output layer.
- Bayesian shrinkage toward position priors.
- Anything correlated across players (stacking, game scripts).
- Bye weeks (any player on bye for the upcoming week should
  arguably be excluded by the pipeline before scoring, but v1 may
  just surface the bye in `notes` and let the user filter).

These are all fair game for the smarter model.
