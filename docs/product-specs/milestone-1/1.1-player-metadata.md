# PRD 1.1 — Player metadata

## Endpoint

`GET https://api.sleeper.app/v1/players/nfl`

Returns a JSON object keyed by Sleeper `player_id`. Roughly 5MB.
Sleeper documents that this should be fetched at most once per day;
we fetch it once per snapshot run, which is at most weekly during
the season.

## Fields we depend on downstream

These fields are load-bearing for the decision engine. If Sleeper
drops or renames one, the loader must fail loudly rather than
producing a "successful" but useless snapshot.

| Field | Why |
| -- | -- |
| `player_id` | Snapshot key. Joins stats + projections + rosters. |
| `full_name` | Display. |
| `position` | Slot eligibility (QB vs WR vs …). |
| `fantasy_positions` | Multi-slot eligibility (some players list `["RB", "WR"]`). |
| `team` | For `--prefer-team` / `--avoid-team` and bye-week handling. |
| `status` | Active / Injured Reserve / etc. Used to skip non-playable. |
| `injury_status` | Surfaced in CLI output. |

Other fields are stored as-is but the loader makes no claim about
them.

## Validation

- Response must be a JSON object with ≥1000 entries (sanity check
  against partial responses — the real one is ~10k+).
- For each entry: `player_id` is present and string-typed. Entries
  missing a `position` are logged and **kept** (defenses, retired
  players, etc. — the decision engine filters them).
- If any of the load-bearing fields are missing from *every* entry,
  abort with "did Sleeper change the schema?"

## Output

Written verbatim to `players.json` in the snapshot folder. No
transformation. The decision engine indexes it on read.

## Failure modes

- HTTP non-200 → retry with exponential backoff up to 3 attempts. If
  still failing, abort the run. Do not write a partial snapshot.
- Response body not parseable as JSON → abort.
- Response object has <1000 entries → abort with "Sleeper returned
  too few players; possible partial response or schema change."
- Load-bearing field universally missing → abort with a clear
  schema-change error.

All abort paths leave no partial snapshot visible to the decision
engine (see PRD 1.3 — atomic write protocol).
