# PRD 1.2 — Weekly stats + projections

## Endpoints

- `GET /v1/stats/nfl/regular/<season>/<week>` — actual stats for a completed week.
- `GET /v1/projections/nfl/regular/<season>/<week>` — Sleeper's own projection for an upcoming or current week.
- `GET /v1/stats/nfl/regular/<season>` — full-season stats for the prior season, used to bootstrap variance estimates before week 1 of the new season.
- `GET /v1/state/nfl` — used to determine the current NFL season and week (so the loader doesn't have to guess).

**The first three are not in the official Sleeper docs.** They are
known to work and are recorded in
[`docs/references/sleeper-api.md`](../../references/sleeper-api.md).
If any of them stop working we will need to revisit.

## What we fetch on each run

1. Call `/v1/state/nfl` → `season`, `week` (the current week — could
   be in-progress).
2. Define `completed_through = week - 1` (the most recently
   completed week of the current season). `0` means we're in
   week 1 or earlier.
3. For each `w` in `1..completed_through`: fetch
   `stats_week_<w>` and `projections_week_<w>`.
4. Also fetch `projections_week_<current_week>` (the upcoming or
   in-progress week). This is what the decision engine compares its
   own projections against.
5. If `completed_through == 0`: fetch `/v1/stats/nfl/regular/<season - 1>`
   as `stats_prior_season.json` for variance bootstrapping.

A CLI flag `--season <year>` and `--week <n>` override the live
state lookup, useful for replaying past weeks or testing against
fixtures.

## Schema (per-week file)

The response is a JSON object keyed by Sleeper `player_id`, with
each value being a flat object of stat codes (`pass_yd`, `rush_td`,
`rec`, etc.) mapped to numeric values. We write it verbatim. The
decision engine multiplies these directly against the league's
`scoring_settings` (which uses the same stat-code keys), so we
don't need to translate.

## Why store both stats and projections

- **Stats** drive the rolling-mean and rolling-stddev variance
  calculations (the "did this player actually score points?" data).
- **Projections** are stored as a sanity check / baseline — the
  buddies want to compare our naive model's output against
  Sleeper's built-in numbers to know whether the model is doing
  anything interesting.

## Validation

- Each response is a JSON object. Empty object is acceptable for
  the *current in-progress* week only. Abort if any past week
  comes back empty — that signals an upstream change.
- Validate the season/week numbers in our request are reflected
  somewhere reasonable in the response if Sleeper echoes them;
  otherwise trust the URL.

## Failure modes

Same retry policy as 1.1: exponential backoff, 3 attempts, then
abort. Individual weeks failing means the whole snapshot is
aborted — partial snapshots break the "latest folder is the source
of truth" invariant.

## Open questions

- Does Sleeper's `regular` path return data for the playoffs? We
  default to regular season only for now. If a league spans
  postseason, we'll need a follow-up.
- Preseason week numbers — out of scope. The decision engine
  doesn't try to project preseason games.
