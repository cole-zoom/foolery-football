# PRD 1.3 — Local storage layout

## Path scheme

```
data/snapshots/<YYYY-MM-DD>[-N]/
├── manifest.json
├── players.json
├── stats_week_<W>.json            # one per completed week of the current season
├── projections_week_<W>.json      # one per available week (past + upcoming)
└── stats_prior_season.json        # only when bootstrapping
```

- `<YYYY-MM-DD>` is the wallclock date the loader **finished**
  writing the snapshot, in the local timezone.
- `[-N]` is appended if a folder for today already exists (e.g.
  `2026-09-15-2` for the second run that day). `N` starts at `2`.
- The decision engine picks the lexicographically last entry in
  `data/snapshots/`. `2026-09-15-2` sorts after `2026-09-15`, which
  is the desired behaviour.

## manifest.json

```json
{
  "schema_version": 1,
  "loader_version": "0.1.0",
  "snapshot_started_at": "2026-09-15T08:00:00-04:00",
  "snapshot_finished_at": "2026-09-15T08:00:42-04:00",
  "season": 2026,
  "completed_through_week": 1,
  "weeks_included": [1],
  "upcoming_week_projection": 2,
  "prior_season_bootstrapped": false,
  "sources": {
    "players": "https://api.sleeper.app/v1/players/nfl",
    "stats_week_1": "https://api.sleeper.app/v1/stats/nfl/regular/2026/1",
    "projections_week_1": "https://api.sleeper.app/v1/projections/nfl/regular/2026/1",
    "projections_week_2": "https://api.sleeper.app/v1/projections/nfl/regular/2026/2"
  }
}
```

`schema_version` is bumped any time we make a non-additive change
to the snapshot format. The decision engine checks this on read
and refuses to interpret a snapshot whose schema version is newer
than the version it understands.

## Write protocol (atomic-ish)

1. Resolve target folder name: `<date>` if free, otherwise `<date>-2`,
   `<date>-3`, etc.
2. Write all files to `data/snapshots/.tmp-<date>-<pid>/`.
3. Write `manifest.json` **last** — it acts as the commit marker.
4. `os.rename` the temp folder to the final name. (`os.rename` is
   atomic on the same filesystem on POSIX, which is what we care
   about for local-only operation.)

A crash mid-write leaves a `.tmp-...` folder. On the next run, the
loader removes any leftover `.tmp-...` folders before starting.
Crashes never leave a half-written snapshot visible to the
decision engine, because the rename is the only thing that makes
the folder discoverable.

## Why immutable, why date-named

- **Reproducible decisions.** A user can re-derive a past
  recommendation by pointing the decision engine at an older
  snapshot.
- **Trivial diffing.** `diff -r data/snapshots/2026-09-15 data/snapshots/2026-09-22`
  shows what changed week over week.
- **No "is this fresh?" guesswork.** Latest folder wins. No mtime
  comparisons, no DB.

## Retention

No automatic cleanup for v1. The user prunes manually. We may add a
`stats-loader prune --keep-last 4` command later if disk usage
becomes annoying.

## Gitignore

`data/snapshots/` is in `.gitignore`. Snapshots are large,
regenerable, and frozen views of third-party data — they don't
belong in version control.
