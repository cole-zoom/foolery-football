# PRD 3.1 — Projections plumbing + availability gate

The snapshot store already holds Sleeper's weekly stat-level
projections (`projections_week_<W>.json`, all seasons 2021–2025, 19
files each). Nothing reads them. This PRD loads them into
`SnapshotData`, extends the leakage contract, and adds a
model-agnostic availability gate to the pipeline.

## Why this is leakage-safe

Sleeper publishes week-W projections **before kickoff**; the archive
serves them as historical record. Using week-W projections to pick a
week-W lineup is exactly what a human does on Sunday morning. The
leakage contract becomes:

> Predicting week W, a model may see: weekly stats for weeks **< W**,
> and projections for weeks **≤ W**.

(See §Validation for the check that the archive really is
pre-kickoff data.)

## Data shape

`projections_week_<W>.json` maps `player_id -> {stat_code: float}`
using the same stat codes as the stats files (`rush_yd`, `rec`,
`rec_tgt`, …) plus Sleeper's precomputed `pts_ppr`/`pts_half_ppr`/
`pts_std` and ADP noise. Because projections are **stat-level**, the
existing `weekly_points(stats, scoring)` scores them under any
league's custom rules — never use the precomputed `pts_*` fields for
scoring (only, optionally, for sanity checks).

**Availability signal:** a player Sleeper does not expect to play has
no meaningful entry — either absent from the table or an entry with
no `gp` key (e.g. only `adp_dd_ppr`). Treat `gp` ≥ 0.5 as "expected
to play".

## Changes

### `types.py` — `SnapshotData`

New field, same shape as `weekly_stats`:

```python
weekly_projections: dict[int, dict[str, dict[str, float]]] = Field(default_factory=dict)
```

No `schema_version` bump: the files already exist in every snapshot;
a reader that loads extra existing files is backward compatible.
Missing files (hypothetical old snapshot) → empty dict for that week.

### `clients/snapshot_reader.py`

Load every `projections_week_<W>.json` present in the season dir into
`weekly_projections`, coercing values with the same `_coerce_stats`
path used for stats. Weeks are discovered from the manifest's
`weeks_included` plus `upcoming_week_projection` (the loader already
snapshots the upcoming week's projection).

### `core/pipeline.py` — trim + gate

- `_snapshot_as_of(snapshot, week)` additionally trims
  `weekly_projections` to weeks **≤ week** (stats stay < week).
- New pool filter beside `_plays_in_week`, same quarantine-over-drop
  convention:

```python
def _projected_to_play(player_id: str, week_projections: dict[str, dict[str, float]] | None) -> bool:
    """False only when the projection table positively says 'out'.

    ``week_projections`` is the target week's table, or None when the
    snapshot has no projections for that week (old snapshot, offseason)
    — unknown means startable, same as the bye filter.
    """
    if week_projections is None:
        return True
    entry = week_projections.get(player_id)
    return entry is not None and entry.get("gp", 0.0) >= 0.5
```

Applied in the `eligible` comprehension in `pipeline.run`, so it
affects every model and every pool (`roster`/`waivers`/`both`)
identically, live decisions and replays alike.

**Week-1 note:** week-1 projections exist (Sleeper projects from
depth charts), so the gate works even when scoring falls back to the
prior season's stats.

### DEF/K

Team defenses and kickers have projection entries. If spot-checks
show `gp` missing for DEFs even when projected (verify during
implementation), scope the `gp` requirement to positions where it's
reliable and fall back to "entry exists" for the rest — the gate must
never disqualify a whole position class.

## Validation (the leakage check)

Extend `scripts/backtest-models.py` with a pseudo-model `sleeper`
that predicts `weekly_points(projection[W][pid], PPR_SCORING)`
directly. Expectations:

- Sleeper-projection MAE should be **better than naive but far from
  zero** (honest weekly forecasting sits around 5–7 MAE for
  startable players). MAE below ~3 for startable players ⇒ suspect
  post-hoc back-fill ⇒ **halt milestone 3.2 and investigate**.
- This pseudo-model also becomes the floor 3.2 must beat.

## Tests

- Reader: season fixture with two projection files → correct
  `weekly_projections`; missing files → `{}`.
- `_snapshot_as_of`: projections trimmed to ≤ W while stats stay < W.
- Gate: entry with `gp: 1.0` startable; absent entry / `gp: 0.0` not;
  `week_projections=None` startable (old snapshot).
- Replay-level: a high-history player with no week-W projection entry
  must not be picked; the human who started him still gets his actual
  points (the gate applies to the model's pool, never to scoring the
  human's real lineup).

## Acceptance

- All existing tests green (the gate can change model picks in
  fixtures only if the fixture explicitly sets projections; fixtures
  without projections behave exactly as before — that is the
  backward-compatibility contract).
- Eval rerun (all three existing models, frozen sample): lineup
  over-prediction bias drops materially (target ≤ +8/wk from +13–22)
  and no model's weekly win rate regresses.

## Out of scope

- Any model-side use of projections (that's 3.2).
- Injury-status string handling (`players.json` `injury_status` is
  snapshot-time state, not week-W state — using it in replay would be
  actual leakage; the projection entry is the historically correct
  signal).
