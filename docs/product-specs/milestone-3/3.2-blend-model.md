# PRD 3.2 — Blend model (`blend`)

New scoring model: the context (ridge) backbone blended with the
week-W Sleeper projection. The projection carries the news signal
(availability, depth charts, matchups) that trailing stats cannot;
the history carries the player-specific calibration that a generic
projection misses. Blend weights shift toward the projection exactly
when history is unreliable.

Depends on 3.1 (`weekly_projections` in the snapshot, gate active).

## Contract

Same `ScoreFn` factory contract as every model (PRD 2.2), registered
as `"blend"` in `scoring.MODELS`. No protocol change: the factory
closure captures the **derived** projection-points table for the
target week (never the raw `SnapshotData` — the existing
`ScoreModelFactory` cache rule).

Target week resolution inside `build(snapshot)`: the trimmed snapshot
(3.1) carries projections for weeks ≤ W and stats for weeks < W, so
the target week is `max(snapshot.weekly_projections)` when present.
When the snapshot has no projections (old snapshot, tests), `blend`
degrades to exactly context.

## Algorithm

Per player, per league scoring dict (lazily fitted, same
`fits_by_league` pattern as context):

1. **History estimate** `m_hist, s_hist` = context's projected mean
   and spread for the player (unchanged code path, reused not
   copied — expose context's internal estimate function if needed).
2. **Projection estimate** `m_proj` =
   `weekly_points(projection_W[player_id], league_scoring)` — the
   stat-level projection scored under the league's own rules.
3. **Blend weight** `w = n / (n + k)` where `n` = number of
   current-season weeks in the player's history and `k = 4.0`
   (weeks of pseudo-history the projection is worth):
   - week 1 / rookie (`n = 0`): 100% projection,
   - `n = 4`: 50/50,
   - `n = 12+`: mostly history, projection still nudging.
4. **Mean** `m = w · m_hist + (1 − w) · m_proj`.
5. **Spread** stays `s_hist` (Sleeper publishes no uncertainty; do
   not invent one). Confidence: bump one level up from context's when
   a projection entry exists (`low → medium`, `medium → high`) — the
   projection is independent evidence.
6. **Risk adjustment** unchanged: `common.risk_adjust(m, s, risk)`.
7. **Notes**: append `"blend: {w:.0%} history / {1−w:.0%} sleeper
   proj {m_proj:.1f}"` so the CLI/web explain the number.

Edge cases:

- No projection entry for the player (gate normally removes them,
  but `pool` callers may score arbitrary players): fall back to pure
  context, note `"no weekly projection"`.
- Projection entry exists but history is empty **and** context would
  return the zero/low fallback: use `m_proj` with position-bucket
  spread (naive's prior) — this replaces the `mean = 0, conf = low`
  dead-end that made rookies unstartable.
- QB/K/DEF: context falls back to naive for these; blend applies the
  same formula on top of naive's estimate — projections are the main
  upgrade at exactly these positions.

## Tuning

`k` is the only free parameter. Fit it once by grid (`k ∈ {2, 4, 6, 8}`)
on the **2024** season backtest (never on the 2025 eval sample — that
is the held-out ship-gate set), pick by startable-MAE, hardcode the
winner with a comment. No config knob.

## Tests

Unit (`tests/unit/test_scoring_blend.py`, fixture style of
`test_scoring_context.py`):

- `n = 0` → pure projection; `n` large → converges to context.
- Projection scored under *league* scoring (custom bonus dict), not
  `pts_ppr`.
- Missing projection entry → identical to context output.
- Rookie with projection → startable mean, not zero.
- Determinism: same inputs, same output (registry contract).

## Acceptance (ship gate, measured per PRD 3.4)

On the frozen 100-league sample: weekly win rate ≥ 40%, mean season
margin ≥ −75, over-prediction bias ≤ +5/wk, and backtest
startable-MAE ≤ both context's and the raw `sleeper` pseudo-model's.
If blend cannot beat the raw projection, the backbone is adding
nothing — ship the simpler thing instead and say so.
